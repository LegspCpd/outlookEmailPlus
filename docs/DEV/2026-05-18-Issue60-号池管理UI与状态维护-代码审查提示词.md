## 审查清单

### 1. 架构与分层

检查是否：

1. 路由、控制器、服务、仓储分层清晰
2. 没有在 Controller 直接写 SQL
3. 没有把业务规则散落在前端直接硬编码

### 2. 查询能力

检查是否：

1. 新增了独立内部管理查询接口
2. 支持池内 / 池外筛选
3. 返回必要的池状态字段
4. 空结果不会报错

### 3. 动作规则

重点检查：

1. `NULL -> available` 是否正确
2. `available -> NULL` 是否正确
3. `restore_available / freeze / retire` 是否只允许在文档约定状态集合内执行
4. `claimed` 是否被通用动作拒绝

### 4. `claimed` 保护（最高优先级）

必须明确回答：

1. `claimed` 是否不能被 `move_out_of_pool`
2. `claimed` 是否不能被 `restore_available`
3. `claimed` 是否不能被 `freeze`
4. `claimed` 是否不能被 `retire`
5. 若实现了 `force_release`，它是不是独立受控动作，而不是偷放行

### 5. 旧接口兼容性

必须检查：

1. `GET /api/accounts` 有没有被污染
2. `/api/external/pool/*` 有没有被破坏
3. 现有概览页统计接口有没有被副作用影响

### 6. 前端交互

检查是否：

1. 有独立入口
2. 有独立页面或清晰管理区域
3. `claimed` 行在 UI 上有明确保护或禁用反馈
4. 没有把危险动作藏进不透明交互

### 7. 测试覆盖

检查是否至少覆盖：

1. 池内 / 池外查询
2. 移入号池
3. 移出号池
4. `claimed` 通用动作拒绝
5. （若实现）`force_release`
