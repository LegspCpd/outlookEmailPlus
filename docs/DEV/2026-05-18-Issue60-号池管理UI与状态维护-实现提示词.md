## 实施顺序（必须按阶段推进）

### Phase 1：内部查询接口 MVP

目标：新增独立查询接口，不污染现有 `GET /api/accounts`

建议接口：

- `GET /api/pool-admin/accounts`

最低返回能力：

1. 支持池内 / 池外筛选
2. 支持 `pool_status` / provider / group / search / 分页
3. 返回 `pool_status`、`claimed_by`、`claimed_at`、`lease_expires_at`、`last_result`

### Phase 2：单账号管理动作 MVP

目标：实现通用状态维护动作

建议动作：

1. `move_into_pool`
2. `move_out_of_pool`
3. `restore_available`
4. `freeze`
5. `retire`

动作规则必须按文档约束：

- `NULL -> available`
- `available / cooldown / used / frozen / retired -> NULL`
- `cooldown / used / frozen / retired -> available`
- `claimed` 禁止走通用动作

若你需要进一步压缩首轮开发批次，可先保证：

1. `move_into_pool`
2. `move_out_of_pool`
3. `claimed` 保护

再继续补 `restore_available / freeze / retire`。

### Phase 3：前端号池管理页 MVP

目标：让管理员真的能看到和操作

最低页面能力：

1. 独立入口
2. 筛选栏
3. 列表区
4. 单账号动作按钮
5. `claimed` 状态禁用不安全动作

### Phase 4：增强项（仅在 MVP 完整后）

若你已完成 MVP 且风险可控，再做：

1. `force_release`
2. 审计
3. 最近流转展示

---

## `claimed` 保护规则（高优先级）

这是本需求最重要的业务保护线：

1. `claimed` 不能被 `move_out_of_pool`
2. `claimed` 不能被 `restore_available`
3. `claimed` 不能被 `freeze`
4. `claimed` 不能被 `retire`
5. 若实现 `force_release`，它必须是单独动作，不得从通用状态更新逻辑偷放行
